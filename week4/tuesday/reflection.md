# Week 4 Tuesday: Reflection

## Question 1: The State File as a System

When you run `terraform apply`, Terraform calls the AWS API to create the resource. AWS responds with the full details of what it created, including the instance ID, private IP, launch time, root volume ID, and many other attributes that were never in the `.tf` config. Terraform takes that full API response and stores it in the state file. So the state file is not built from your configuration. It is built from what the cloud provider reports back after creating or refreshing the resource.

When you run `terraform destroy`, Terraform reads the state file to find every resource it manages, calls the API to delete each one, and then removes them from the state file. After a successful destroy, the state file is essentially empty (it still exists, but it lists no resources).

If you deleted the state file manually without destroying the infrastructure, Terraform would have no memory of the resources it created. On the next `terraform plan`, Terraform would see the resources declared in your `.tf` files, find nothing in state, and propose to create everything from scratch. But the old resources would still be running in AWS, so you would end up with duplicate infrastructure. The old instances, security groups, and other resources would become orphans that you would have to clean up manually through the AWS console or CLI.

The correct recovery procedure is to use `terraform import` to bring existing resources back into a new state file. For example, `terraform import aws_instance.kk_api i-0abc123def456` would tell Terraform "this instance already exists, track it." You would need to import every resource that still exists in the cloud. After importing, you should run `terraform plan` to confirm the config and state match with no changes needed.

## Question 2: The (known after apply) Values

Some values are shown as `(known after apply)` in the plan because they do not exist yet. They are generated by AWS at creation time, and Terraform has no way to predict them.

**Example 1: `public_ip`**. The public IP address of the EC2 instance is assigned by AWS when the instance launches. Terraform cannot know this value at plan time because the IP comes from AWS's pool of available addresses, and the instance does not exist yet.

**Example 2: `id` (the instance ID)**. The instance ID like `i-0abc123def456` is generated by AWS when the instance is created. There is no way to predict what ID AWS will assign, so Terraform marks it as `(known after apply)`.

If an output value depends on a `(known after apply)` attribute, Terraform shows that output as `(known after apply)` as well during `terraform plan`. For example, the `instance_public_ip` output in our config references `aws_instance.kk_api.public_ip`, so during plan it would show `(known after apply)` instead of an actual IP address. The real value only appears after `terraform apply` completes.

## Question 3: Hardcoded vs Variable

In our configuration, the SSH source IP is actually already a variable (`var.my_ip_address`), but the value is set in `terraform.tfvars` as `"105.160.80.49/32"`. This is better than hardcoding it directly in `main.tf`, but it is still a problem for a shared team configuration because that `.tfvars` file contains one person's IP address. If a teammate clones this repo and runs `terraform apply`, they would either need to edit the `.tfvars` file with their own IP, or they would lock themselves out of SSH because the security group only allows the original IP.

The solution is to make the variable accept a list of CIDR blocks instead of a single string. The variable type would be `list(string)`:

```hcl
variable "allowed_ssh_cidrs" {
  description = "List of CIDR blocks allowed to SSH"
  type        = list(string)
}
```

Then in the security group ingress rule, you would use `cidr_blocks = var.allowed_ssh_cidrs` instead of wrapping a single value in brackets. Each team member's IP (or the office IP range) could be listed in the `.tfvars` file:

```hcl
allowed_ssh_cidrs = ["105.160.80.49/32", "203.0.113.10/32"]
```

This way the configuration supports multiple engineers without anyone editing the Terraform code itself.

## Question 4: What Tuesday's Configuration Cannot Do

For a production deployment, the following values would need to change:

- **instance_type**: production would need something larger than `t3.micro`, like `t3.medium` or `m5.large`
- **environment tag**: would change from `"staging"` to `"production"`
- **instance_name**: would need a different name to distinguish it from staging
- **VPC**: production should use a dedicated VPC with private subnets, not the default VPC
- **SSH access**: production might restrict SSH to a bastion host or VPN range instead of a personal IP
- **number of instances**: production likely needs multiple VMs behind a load balancer for availability
- **region**: production might deploy to multiple regions for redundancy

With today's single-file approach, you would handle this by changing the variable values in `terraform.tfvars` and running `terraform apply` again. But this is where things break: Terraform only tracks one state file. If you change the instance type from `t3.micro` to `t3.medium` and apply, Terraform does not create a new production VM alongside staging. It destroys the staging VM and replaces it with the new one. You cannot have both environments running at the same time from one configuration with one state file.

If production needs a different VPC or a different number of VMs, you would have to either duplicate the entire configuration directory (which leads to drift between copies) or manually swap `.tfvars` files and manage separate state files yourself. Both approaches are error-prone. You could accidentally destroy staging by running apply with the wrong variables, or forget to update one of the copies when the base configuration changes.

This is the problem that Wednesday solves with workspaces, modules, and environment-specific variable files, so you can manage multiple environments from a shared configuration without them interfering with each other.
